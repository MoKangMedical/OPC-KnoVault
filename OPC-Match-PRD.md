## 1. 一句话定位

OPC Match 是 AI 时代的一人公司可信商业合作网络。

平台帮助一人公司 OPC 将技能、知识库、数据集和商业资源结构化为可验证商业资产，并通过 AI Agent 自动匹配合作机会。Monad 用于记录关键资产确权、授权和合作意向证明。

## 2. 大白话版本

一个人也可以像公司一样挂牌自己的技能、知识库和数据资产。需求方发布商业机会后，AI 帮他找到合适的一人公司或临时 OPC 小队，Monad 记录关键合作证明和资产归属。

## 3. 产品背景

AI 时代，个人的商业能力正在从“简历上的经验”变成可复用、可组合、可交易的资产，例如：

- 技能：开发、增长、设计、运营、BD、研究、咨询。
- 知识库：行业 SOP、市场进入 Playbook、专家经验库。
- 数据集：社区数据、增长数据、行业联系人图谱、用户洞察。
- Agent 能力包：Prompt、工作流、自动化任务链、垂直 Agent。
- 商业资源：渠道、社群、KOL、客户线索、项目协作机会。

传统招聘平台偏向全职雇佣，外包平台偏向任务接单，都不足以表达“一个人像公司一样交付”的新形态。OPC Match 的目标是让 OPC 的能力被结构化展示、智能匹配、可信证明，并逐步沉淀成可验证的商业履历。

## 4. 目标用户

### 4.1 需求方

需求方是 MVP 的主用户，Demo 从需求方视角开场。

典型用户包括：

- Web3 项目方。
- Monad 生态项目。
- AI 创业团队。
- 传统企业创新部门。
- 需要外部专家能力的小型团队。

他们的核心诉求是：快速找到可信、可验证、能独立交付的一人公司或 OPC 小队。

### 4.2 供给方

供给方是 OPC，也就是可以像公司一样独立交付商业结果的个人或小型协作单元。

典型 OPC 包括：

- 独立开发者。
- AI 产品顾问。
- Web3 增长专家。
- 社区运营者。
- BD / GTM 顾问。
- 数据分析师。
- 行业知识专家。
- 拥有知识库、数据集或 Agent 能力包的个人。

## 5. 核心问题

### 5.1 需求方痛点

- 很难找到可信的一人公司合作方。
- 传统招聘太重，外包平台缺少可信证明。
- AI 时代的能力越来越碎片化，技能、知识库、数据集、Agent 工作流都可能成为交易对象。
- 跨地域、跨组织合作缺少低成本的意向证明、授权证明和声誉沉淀。
- 找人过程依赖熟人推荐，效率低，难以规模化。

### 5.2 OPC 痛点

- 个人能力难以像公司一样被结构化展示。
- 知识库、数据集、Agent 能力包难以确权和授权。
- 历史合作和声誉难以积累为可验证资产。
- 获客依赖社交网络，缺少 AI 自动撮合渠道。
- 临时组队承接更大项目时，缺少可信协作记录。

## 6. 产品目标

### 6.1 长期目标

建立一个面向一人公司的可信商业机会网络，让 OPC 可以挂牌资产、获得机会、完成协作，并积累链上可信履历。

### 6.2 黑客松 MVP 目标

在黑客松中完成一个可点击、可讲清楚、可真实上链的核心闭环：

需求方发布 Web3 中国市场进入机会，AI 推荐可信 OPC 或 OPC 小队，需求方确认合作意向，并在 Monad 上写入 Intent Proof。

## 7. 黑客松成功标准

优先级如下：

1. Demo 闭环完整。
2. Monad / Web3 特色清楚。
3. AI Agent 体验明显。
4. 商业故事有想象空间。

验收标准：

- 用户能从机会发布走到 AI 推荐。
- 用户能看到可信 OPC 档案。
- 用户能选择 OPC 并生成合作意向。
- 用户能在 Monad 上完成 Intent Proof。
- 页面能展示链上证明结果、交易哈希和证明状态。

## 8. 产品设计原则

### 8.1 Monad-native

OPC Match 不是把钱包按钮贴到普通 SaaS 上，而是围绕可信证明设计核心流程。

MVP 中 Monad 负责记录：

- 合作意向证明。
- 参与方地址。
- 机会 ID。
- 匹配资产摘要。
- AI 匹配摘要哈希。
- 合作状态。

详细商业内容、报价、沟通记录和资产详情放在链下，链上只保存可验证的关键状态与哈希。

### 8.2 并行 EVM 叙事

OPC Match 中会产生大量独立、小额、高频的商业证明，例如：

- 资产注册。
- 资产授权。
- 合作意向。
- 交付确认。
- 声誉评价。

这些证明之间大多相互独立，天然适合高吞吐、低延迟、并行执行的 EVM 环境。Monad 的并行 EVM 可以支撑这种“高频可信商业事件网络”的未来形态。

### 8.3 Agentic X

AI Agent 不只是聊天机器人，而是完成一条行动链：

理解机会，推荐 OPC，解释可信依据，提示风险，生成合作意向摘要，准备链上证明，最后由用户审核并签名。

### 8.4 Web3 应用快速构建

MVP 采用轻链上、重体验的方式：

- 前端优先展示完整业务闭环。
- 使用预置 OPC 和机会数据保证 Demo 稳定。
- 使用真实钱包连接和 Monad 写入证明体现 Web3 可信闭环。
- 不在 MVP 中处理复杂支付、仲裁和大规模市场治理。

## 9. MVP 主场景

Demo 主场景：

一个海外 Web3 / Monad 生态项目希望进入中国和上海市场，需要找到可信 OPC 或 OPC 小队，完成本地增长、社区冷启动、BD 资源连接、内容传播和市场数据分析。

机会名称示例：

Monad 生态增长伙伴招募：中国 / 上海市场进入

需求方目标：

- 进入华语 Web3 社区。
- 建立上海本地 Builder / VC / KOL 连接。
- 完成社区冷启动。
- 产出本地化内容。
- 获得市场数据和增长建议。

## 10. 机会模板

MVP 产品范围包含四类机会模板：

1. Web3 市场进入，Demo 主模板。
2. AI 增长 / 获客。
3. 企业内部 AI Agent / 知识库。
4. 创业团队 MVP 小队。

四类模板共享同一套流程：

选择机会模板，填写目标和约束，AI 匹配 OPC / 资产 / 小队，确认合作意向，写入 Monad 证明。

## 11. MVP 用户流程

1. 需求方进入 OPC Match 工作台。
2. 需求方连接钱包。
3. 需求方选择“Web3 市场进入”机会模板。
4. 需求方填写项目名称、目标市场、预算区间、周期和所需能力。
5. AI Agent 基于规则和预置数据生成推荐结果。
6. AI 展示推荐 OPC、推荐理由、可信依据、风险提示和合作建议。
7. 需求方查看 OPC 可信档案。
8. 需求方选择一个 OPC 或 OPC 小队。
9. 系统生成合作意向摘要和摘要哈希。
10. 需求方确认并签名。
11. 系统在 Monad 上写入 Intent Proof。
12. 平台展示链上证明结果页。

## 13. OPC 可信档案结构

每个 OPC 档案包含：

- 身份与定位：OPC 名称、个人简介、所在城市、时区、钱包地址。
- 技能资产：技能标签、代表案例、可交付服务。
- 知识资产：行业 Playbook、SOP、专家知识库。
- 数据资产：数据集名称、数据范围、授权方式。
- Agent 能力包：Prompt、工作流、自动化任务链。
- 可信证明：历史合作、链上证明、评价、资产确权记录。
- 合作偏好：预算区间、可接项目类型、交付周期、合作方式。

MVP 页面重点展示：

- 技能资产。
- 知识 / 数据 / Agent 资产。
- 可信证明。

## 14. AI Agent 能力

### 14.1 MVP 能力

AI Agent 在 MVP 中需要完成：

- 根据机会标签、行业、预算、地区、所需能力推荐 OPC。
- 对候选 OPC 生成匹配理由。
- 标注可信证据。
- 提示合作风险。
- 给出合作建议。
- 生成合作意向摘要。
- 帮用户准备链上证明内容。

### 14.2 实现策略

MVP 使用稳定优先的方式：

- 匹配结果基于规则和预置数据。
- 推荐解释、风险提示、合作建议由 LLM 生成或预置为可控文本。
- Agent 准备链上证明字段，用户最终签名确认。

### 14.3 暂不做

- 全自动检索真实互联网数据。
- 全自动替用户发起链上交易。
- 复杂多轮谈判。
- 自动付款托管。
- 真实 KYC。

## 15. Monad 链上设计

### 15.1 MVP 链上对象

MVP 优先实现标准合作意向证明：Intent Proof。

### 15.2 Intent Proof 字段

Intent Proof 记录字段：

- 需求方钱包地址。
- OPC 钱包地址，或 OPC 小队成员地址。
- 机会 ID。
- 机会类型。
- 匹配技能。
- 匹配知识库 / 数据集 / Agent 能力包。
- 预算区间。
- 合作状态。
- AI 匹配摘要哈希。
- 时间戳。

### 15.3 链上与链下边界

链上保存：

- 参与方地址。
- 机会 ID。
- 机会类型。
- 匹配资产摘要。
- 状态。
- 哈希。
- 时间戳。

链下保存：

- 完整机会描述。
- AI 推荐解释。
- 商业报价。
- 沟通记录。
- 详细资产内容。
- 数据集或知识库实际内容。

### 15.4 后续可扩展链上对象

- OPC 资产确权证明。
- 知识库 / 数据集授权证明。
- 交付确认证明。
- 合作评价证明。
- 声誉 SBT / NFT。
- 多 OPC 小队协作凭证。

## 16. Demo 脚本

建议 Demo 控制在 4 到 5 分钟。

### 16.1 开场，0:00 到 0:30

话术：

AI 时代，一个人可以用技能、知识库、数据集和 Agent 工作流像公司一样交付商业价值。但今天这些能力很难被发现、匹配和验证。OPC Match 是面向一人公司的可信商业合作网络。

### 16.2 发布机会，0:30 到 1:20

演示动作：

- 进入机会发布页。
- 连接钱包。
- 选择 Web3 市场进入模板。
- 输入示例：Monad 生态项目希望进入中国 / 上海市场。
- 选择能力：社区冷启动、BD、本地内容、增长分析。
- 点击生成匹配。

讲述重点：

需求方不是招全职员工，而是寻找能快速交付的一人公司或临时 OPC 小队。

### 16.3 AI 推荐 OPC，1:20 到 2:20

演示动作：

- 进入 AI 匹配结果页。
- 展示 3 个推荐 OPC。
- 展示 AI 推荐理由。
- 展示风险提示。
- 展示推荐小队组合。

讲述重点：

Agent 不只是搜索标签，它会解释为什么匹配、哪些资产可用、合作风险在哪里，以及建议怎样开始合作。

### 16.4 查看可信档案，2:20 到 3:10

演示动作：

- 进入某个 OPC 档案页。
- 展示技能资产：Web3 增长、社区运营、华语内容。
- 展示知识资产：中国 Web3 市场进入 Playbook。
- 展示数据资产：华语社区增长数据集。
- 展示链上证明：历史合作、资产确权、授权记录。

讲述重点：

这是 AI 时代的一人公司主页。它不是简历，而是可交易、可授权、可验证的商业资产组合。

### 16.5 生成 Intent Proof，3:10 到 4:10

演示动作：

- 进入合作意向确认页。
- 展示需求方地址、OPC 地址、机会类型、预算区间。
- 展示 AI 匹配摘要哈希。
- 点击签名并写入 Monad。
- 等待交易成功。

讲述重点：

Monad 记录的是关键合作证明，不把商业细节全部上链。这样既保护隐私，又能形成可信、可追溯的合作履历。

### 16.6 结果页收尾，4:10 到 5:00

演示动作：

- 展示交易哈希。
- 展示 Intent Proof ID。
- 展示 Recorded on Monad。
- 回到 OPC 档案，显示新增合作意向证明。

收尾话术：

OPC Match 未来可以扩展到资产确权、授权交易、交付评价、声誉网络和多 OPC 自动组队。Monad 的并行 EVM 适合承载大量独立的小型商业证明，让一人公司之间的合作变得更可信、更自动化。

## 21. 风险与应对

### 21.1 范围过大

风险：

产品同时涉及 AI、Web3、招聘、资产交易和协作网络，黑客松期间容易发散。

应对：

Demo 只跑 Web3 中国市场进入这一条主线，其他机会模板用数据和入口展示，不做完整流程。

### 21.2 AI 匹配不稳定

风险：

现场实时生成推荐可能不稳定。

应对：

候选 OPC 使用预置数据，LLM 主要生成解释和摘要。必要时推荐解释也可以使用预置文本。

### 21.3 链上交互失败

风险：

钱包、测试网、RPC 或合约调用可能现场失败。

应对：

准备链上成功状态截图或预置交易哈希作为备份，但主流程优先使用真实交易。

### 21.4 Web3 价值表达不清

风险：

评委可能认为这是普通 AI 撮合产品。

应对：

在 Demo 中强调 Intent Proof、资产确权、授权和声誉沉淀是平台核心，不是附加功能。
